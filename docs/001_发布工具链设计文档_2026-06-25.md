## 1. 目标与边界

`@usethink/publish-toolkit` 的目标是把 npm 包发布流程集中成一个可复用 CLI，避免每个项目复制一套发布脚本。

它负责：

- 校验包目录、`package.json`、`private` 字段。
- 可选检查 git 工作区是否干净。
- 可选检查 npm registry 中同版本是否已发布。
- 按规则执行构建和发布命令。
- 在正式发布时临时写入 `.npmrc`，并在退出时清理 token。
- 提供构建产物混淆能力。

它不负责：

- 替调用方决定是否应该 bump 版本。
- 替调用方创建 npm token 或 GitHub secret。
- 替调用方绕过 npm 的同版本不可覆盖规则。
- 通过全局环境猜测调用方应使用哪个包管理器。

## 2. 当前关键设计

### 2.1 CLI 启动方式与发布包管理器分离

一个容易混淆的点是：

- `npx --yes -p @usethink/publish-toolkit publish-toolkit ...`、项目本地 `npx publish-toolkit ...` 或 `pnpm exec publish-toolkit ...` 只是启动 CLI 的方式。
- CLI 内部执行 `npm publish` 还是 `pnpm publish` 由 `--package-manager` 决定。

当前默认行为：

```bash
publish-toolkit publish
# 默认等价于：
publish-toolkit publish --package-manager npm
```

包管理器选择规则：

| 参数 | 行为 |
|------|------|
| `--package-manager npm` | 默认值。执行 `npm run build` / `npm publish`。 |
| `--package-manager pnpm` | 显式执行 `pnpm run build` / `pnpm publish --no-git-checks`。 |
| `--package-manager auto` | 只读取项目元数据；`package.json.packageManager` 为 `pnpm@...` 或仅存在 `pnpm-lock.yaml` 时选择 pnpm，否则选择 npm。 |

重要约束：

- `auto` 不检测 runner 全局是否安装了 pnpm。
- 机器上存在 pnpm 不代表项目可安全使用 pnpm 发布。
- 默认 npm 是为了避免 pnpm v11 的 ignored build scripts 策略误伤 npm 项目。

### 2.2 发布引擎流程

```text
publish-toolkit publish
  |
  |-- Step 1: 验证参数
  |   |-- dry-run 可不提供 NPM_TOKEN
  |   |-- 正式发布必须提供 NPM_TOKEN
  |   |-- package.json 必须存在且 private !== true
  |   |-- 解析 package name/version/scripts/packageManager
  |   `-- 解析发布包管理器：npm | pnpm | auto
  |
  |-- Step 2: git 检查
  |   |-- 默认检查 git status --porcelain
  |   `-- --no-git-check 可跳过
  |
  |-- Step 3: 版本检查
  |   |-- npm view <pkg> version
  |   |-- 本地版本已存在则失败，避免覆盖发布
  |   `-- --no-version-check 可跳过
  |
  |-- Step 4: 构建
  |   |-- 有 prepack：跳过手动构建，交给 npm/pnpm publish 生命周期
  |   |-- 有 build：按包管理器执行 <pm> run build
  |   `-- dry-run 只打印，不执行
  |
  |-- Step 5: 认证
  |   |-- dry-run 只打印，不写 .npmrc
  |   |-- 正式发布临时写 .npmrc
  |   `-- exit/SIGINT/SIGTERM 时清理
  |
  `-- Step 6: 发布
      |-- npm publish 或 pnpm publish
      |-- dry-run 添加 --dry-run
      |-- pnpm 添加 --no-git-checks
      `-- 解析 401/403/404/EOTP 错误提示
```

### 2.3 自发布 workflow

本仓库发布 `@usethink/publish-toolkit` 自身时，不使用 toolkit CLI 自举发布，而是直接使用 npm：

```yaml
- run: npm install
- run: npm run type-check
- run: npm test
- run: npm run build
- run: npm publish --tag=latest
```

当前 workflow 还有一个保护步骤：

```bash
npm view "${PACKAGE_NAME}@${PACKAGE_VERSION}" version
```

如果 npm 上已经存在当前版本，workflow 会跳过 `npm publish` 并成功退出。这样允许先手工发布 npm，再推送 tag 做源码追溯，不会因为重复发布同版本产生红灯。

## 4. CLI 命令

### 4.1 publish

```bash
publish-toolkit publish \
  --package <path> \
  --registry <url> \
  --dry-run \
  --tag <name> \
  --otp <code> \
  --access <level> \
  --package-manager <npm|pnpm|auto> \
  --no-git-check \
  --no-version-check \
  --verbose
```

默认值：

| 选项 | 默认值 |
|------|--------|
| `--package` | 当前目录 |
| `--registry` | `https://registry.npmjs.org/` |
| `--dry-run` | `false` |
| `--tag` | `latest` |
| `--access` | `public` |
| `--package-manager` | `npm` |
| git 检查 | 开启 |
| 版本检查 | 开启 |

### 4.2 obfuscate

```bash
publish-toolkit obfuscate run \
  --input ./dist \
  --output ./dist-obf \
  --level light \
  --source-map \
  --exclude "**/*.test.js" \
  --report
```

支持级别：

- `none`：只复制文件。
- `light`：轻度混淆。
- `medium`：中度混淆。
- `aggressive`：强混淆，体积会明显膨胀。

辅助命令：

```bash
publish-toolkit obfuscate list-levels
publish-toolkit obfuscate info
```

## 5. 推荐接入方式

### 5.1 npm 项目

```yaml
- run: npm ci
- run: npm run type-check
- run: npm test
- run: npm run build
- run: npx publish-toolkit publish
  env:
    NPM_TOKEN: ${{ secrets.NPM_TOKEN }}
```

说明：上面的 `npx publish-toolkit publish` 假设项目已经把 `@usethink/publish-toolkit` 安装为 devDependency。未安装时使用：

```yaml
- run: npx --yes -p @usethink/publish-toolkit publish-toolkit publish
  env:
    NPM_TOKEN: ${{ secrets.NPM_TOKEN }}
```

### 5.2 pnpm 项目但希望仍用 npm publish

```yaml
- run: pnpm install --frozen-lockfile
- run: pnpm test
- run: pnpm build
- run: pnpm exec publish-toolkit publish --package-manager npm
  env:
    NPM_TOKEN: ${{ secrets.NPM_TOKEN }}
```

### 5.3 确认项目已经配置好 pnpm 发布策略

只有当项目已经明确处理 pnpm v11 build scripts 策略时，才使用：

```bash
publish-toolkit publish --package-manager pnpm
```

或：

```bash
publish-toolkit publish --package-manager auto
```

`auto` 适合 `package.json` 中有明确声明的项目：

```json
{
  "packageManager": "pnpm@11.9.0"
}
```

## 7. 设计决策记录

### 7.1 默认 npm，不默认 pnpm

决策：`publish-toolkit publish` 默认执行 `npm publish`。

原因：

- npm 是 npm registry 发布的最小公共路径。
- `pnpm exec` 只是 CLI 启动方式，不应污染内部发布命令选择。
- pnpm v11 默认会拦截未批准的 build scripts，消费项目未配置时会失败。

拒绝方案：

- “全局有 pnpm 就用 pnpm”：已在 `mach-calculators` 发布中导致 `ERR_PNPM_IGNORED_BUILDS`。
- “所有消费项目都补 pnpm allowBuilds”：这是消费侧绕行，工具本身仍不可靠。

### 7.2 auto 只看项目元数据

决策：`--package-manager auto` 不检查 `command -v pnpm`。

原因：

- CI runner 全局工具不可作为项目契约。
- 项目元数据才代表维护者意图。

当前 auto 判断：

1. `package.json.packageManager` 以 `pnpm@` 开头。
2. 仅存在 `pnpm-lock.yaml`，且不存在 `package-lock.json`。
3. 其他情况默认 npm。

### 7.3 自发布不用 CLI 自举

决策：publish-toolkit 仓库自身直接 `npm publish`。

原因：

- 避免发布工具用未发布版本发布自己时产生自举复杂度。
- 自发布 workflow 更容易诊断。
- 对外 CLI 行为仍由单元测试和 `mach-calculators` 集成验证覆盖。

## 8. 验证要求

发布工具变更后必须至少执行：

```bash
npm run type-check
npm test
npm run build
npm publish --dry-run --access public --tag latest --registry=https://registry.npmjs.org/
```

涉及发布引擎时，还需要在真实消费项目中验证，例如：

```bash
cd /Users/dingyuwang/0-X/2-node/mach-calculators
npx --yes -p @usethink/publish-toolkit@<version> publish-toolkit publish \
  --dry-run \
  --no-git-check \
  --no-version-check \
  --verbose
```

预期输出必须包含：

```text
包管理器: npm（显式配置）
[dry-run] 将执行: npm publish ...
```

## 9. 维护注意事项

- 修改 `src/` 后必须运行 `npm run build`，让 `dist/` 同步。
- bump `package.json` 后必须同步 `package-lock.json`。
- 推送 tag 前确认 npm 是否已存在该版本。
- 不能把 npm token 写入文件或提交到 git。
- 文档中所有发布示例必须体现“默认 npm，pnpm 显式选择”。
- `mach-calculators` 这类独立 npm 包默认应使用 npm 发布。
