# 发布工具链设计文档

> @usethink/publish-toolkit — 统一 npm 包发布工具链  
> 创建日期：2026-06-25

---

## 1. 背景与问题

### 1.1 现状

在多项目管理中，每个项目都维护了一套独立的发布逻辑：

- `scripts/publish-package.mjs`：本地发布脚本
- `.github/workflows/publish.yml`：CI 自动发布
- 各类构建、检查、npm publish 命令

以 cf-core 为例，发布流程分散在：

```
cf-core/
├── scripts/
│   ├── publish-package.mjs      # 本地发布脚本（500+ 行）
│   └── setup-github-secrets.sh  # Secrets 配置脚本
└── .github/workflows/
    └── publish.yml               # CI 发布 workflow
```

### 1.2 核心痛点

| 痛点 | 表现 |
|------|------|
| **维护灾难** | 10 个项目 = 10 套脚本，任何修改需同步到所有项目 |
| **Bug 修复成本高** | 修复一个发布逻辑问题需改 10 个地方 |
| **新功能扩散慢** | 新增特性（如 dry-run 增强）需逐个项目添加 |
| **CI 配置冗长** | 每个项目的 publish.yml 重复 80% 以上 |
| **源码污染** | 每个项目都有 scripts/ 目录，但发布逻辑不属于业务代码 |

### 1.3 目标

作为个人开发者，最优方案应满足：

1. **一次编写，到处运行**：发布逻辑只维护一份
2. **各项目零脚本**：不需要每个项目都有 scripts/obfuscate.mjs
3. **升级无感知**：工具更新后，各项目自动获得新能力
4. **CI 无副作用**：不污染项目源码目录
5. **简单可用**：个人开发者也能轻松维护

## 2. 构思过程

### 2.1 方案选型

#### 方案 A：继续每项目一套脚本

- 优点：无迁移成本
- 缺点：维护成本随项目数量线性增长，个人开发者负担重
- **结论：不采用**

#### 方案 B：GitHub Actions 共享 Workflow

- 优点：CI 配置复用
- 缺点：仅解决 CI 配置问题，本地发布脚本仍需维护；各项目仍需配置 inputs/outputs
- **结论：不采用**

#### 方案 C：独立 npm 工具包（最终方案）

- 优点：
  - 发布逻辑集中维护
  - 各项目通过 `npx publish-toolkit` 或 `npm install -D` 使用
  - 标准 npm 包生态，无额外基础设施
  - Bug 修复一次，所有项目受益
- 缺点：
  - 需要额外发布一个 npm 包
  - 初次接入有学习成本
- **结论：采用**

### 2.2 设计原则

1. **零侵入**：输入输出均为文件路径，不修改项目源码结构
2. **向后兼容**：保留原 `scripts/publish-package.mjs` 的所有能力
3. **渐进式**：各项目可独立选择接入时机，不强制定制
4. **可扩展**：混淆、打包、发布等能力模块化，便于后续增强

## 4. 内部运行流程

### 4.1 发布引擎流程

```
用户调用
  ↓
npx publish-toolkit publish --dry-run
  ↓
┌─────────────────────────────────────────┐
│ Step 1: 验证参数                         │
│   - 检查 NPM_TOKEN 环境变量              │
│   - 检查 package.json 是否存在            │
│   - 检查 private 是否为 true             │
└─────────────────────────────────────────┘
  ↓
┌─────────────────────────────────────────┐
│ Step 2: git 工作区检查                    │
│   - git rev-parse --show-toplevel        │
│   - git status --porcelain               │
│   - 有未提交变更则报错退出                 │
└─────────────────────────────────────────┘
  ↓
┌─────────────────────────────────────────┐
│ Step 3: 版本号已发布检查                  │
│   - npm view <pkg> version               │
│   - 与本地 package.json 比较              │
│   - 版本已存在则报错退出                  │
└─────────────────────────────────────────┘
  ↓
┌─────────────────────────────────────────┐
│ Step 4: 构建                              │
│   - 检测 prepack 脚本（npm publish 自动执行）│
│   - 或执行 build 脚本                    │
│   - 无构建脚本则跳过                       │
└─────────────────────────────────────────┘
  ↓
┌─────────────────────────────────────────┐
│ Step 5: 配置认证                          │
│   - 写入临时 .npmrc                       │
│   - 包含 registry 和 _authToken           │
│   - 注册进程退出清理钩子                   │
└─────────────────────────────────────────┘
  ↓
┌─────────────────────────────────────────┐
│ Step 6: 执行 publish                      │
│   - 优先使用 pnpm，回退 npm                │
│   - 拼接参数：--registry --access --tag  │
│   - dry-run 模式添加 --dry-run            │
│   - 解析常见错误（404/403/401/EOTP）       │
└─────────────────────────────────────────┘
  ↓
完成 / 清理 .npmrc
```

### 4.2 混淆引擎流程

```
用户调用
  ↓
npx publish-toolkit obfuscate --input ./dist --output ./dist-obf --level light
  ↓
┌─────────────────────────────────────────┐
│ 初始化 Obfuscator                        │
│   - 根据 level 选择策略                  │
│   - none: 直接跳过                        │
│   - light/medium/aggressive: 执行混淆     │
└─────────────────────────────────────────┘
  ↓
┌─────────────────────────────────────────┐
│ 遍历输入目录                              │
│   - 递归扫描 .js/.mjs/.ts 文件           │
│   - 应用混淆规则                          │
│   - 写入输出目录                          │
│   - 生成 source map（可选）               │
└─────────────────────────────────────────┘
  ↓
返回处理结果（processedFiles / outputDir / message）
```

**注意**：当前混淆引擎为占位实现，仅返回模拟结果。真实场景下可替换为 `javascript-obfuscator`。

### 4.3 CLI 参数解析

使用 `commander` 库，支持：

```bash
# 发布命令
publish-toolkit publish \
  --package <path> \
  --registry <url> \
  --dry-run \
  --tag <name> \
  --otp <code> \
  --access <level> \
  --git-check \
  --no-git-check \
  --version-check \
  --no-version-check \
  --verbose

# 混淆命令
publish-toolkit obfuscate \
  --input <dir> \
  --output <dir> \
  --level <name> \
  --source-map
```

## 5. 使用场景

### 5.1 场景一：本地开发时预览发布

开发者修改代码后，想在本地验证发布流程是否正确：

```bash
NPM_TOKEN=npm_xxx npx publish-toolkit publish --dry-run
```

**优势**：
- 无需推送 tag
- 无需触发 CI
- 本地快速验证

### 5.2 场景二：GitHub Actions 自动发布

项目配置 CI，推送 tag 时自动发布：

```yaml
# .github/workflows/publish.yml
- run: npx publish-toolkit publish
  env:
    NPM_TOKEN: ${{ secrets.NPM_TOKEN }}
```

**优势**：
- 标准化发布流程
- 自动执行测试和构建
- 统一的错误处理

### 5.3 场景三：应急手动发布

CI 失败或需要紧急发版时，通过 GitHub UI 手动触发：

```yaml
workflow_dispatch:
  inputs:
    dry_run:
      type: boolean
      default: false
    tag:
      type: string
      default: latest
```

### 5.4 场景四：多项目统一发布标准

公司/团队有多个 npm 包，需要统一的发布标准：

```bash
# 所有项目使用相同的发布逻辑
npx publish-toolkit publish --registry=https://registry.npmjs.org/ --access=public
```

## 6. 使用方式方法

### 6.1 作为 devDependency 安装

```bash
# 安装到项目
npm install -D @usethink/publish-toolkit

# 或使用 pnpm
pnpm add -D @usethink/publish-toolkit
```

### 6.2 直接使用 npx（无需安装）

```bash
# 直接运行，无需安装到本地
npx publish-toolkit publish --dry-run
```

**注意**：根据团队记忆，优先使用 `pnpm exec` 而非 `npx`，避免在某些环境/目录下 `npx` 被解析为 `npm run`。

```bash
# 推荐方式（项目已安装为 devDependency 时）
pnpm exec publish-toolkit publish --dry-run

# 退回到 npx（工具未安装时）
npx publish-toolkit publish --dry-run
```

### 6.3 在 package.json 中配置脚本

```json
{
  "scripts": {
    "publish": "publish-toolkit publish",
    "publish:beta": "publish-toolkit publish --tag beta",
    "publish:dry": "publish-toolkit publish --dry-run"
  }
}
```

使用：

```bash
pnpm run publish
pnpm run publish:beta
pnpm run publish:dry
```

### 6.4 GitHub Actions 集成示例

```yaml
name: Publish

on:
  push:
    tags:
      - 'v*'
  workflow_dispatch:
    inputs:
      dry_run:
        type: boolean
        default: false

jobs:
  publish:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 22
          registry-url: 'https://registry.npmjs.org'
      
      - run: npm install
      - run: npm test
      - run: npm run build
      
      - run: npx publish-toolkit publish
        env:
          NPM_TOKEN: ${{ secrets.NPM_TOKEN }}
```

### 6.5 混淆使用示例

```bash
# 轻度混淆
npx publish-toolkit obfuscate \
  --input ./dist \
  --output ./dist-obf \
  --level light

# 保留 source map
npx publish-toolkit obfuscate \
  --input ./dist \
  --output ./dist-obf \
  --level medium \
  --source-map
```

## 7. 踩坑记录

### 坑 1：npx 在某些环境被解析为 npm run

**现象**：
```bash
npx publish-toolkit publish
# 报错：Missing script: "publish-toolkit"
```

**原因**：
在某些 shell/目录配置下，`npx` 被解析为 `npm run`，而不是执行 `node_modules/.bin` 下的二进制。

**解决方案**：
- 优先使用 `pnpm exec publish-toolkit publish`（项目已安装为 devDependency 时）
- 退回到 `npx` 时确保工具未安装在本地

### 坑 2：Commander 的 --no-* 标志行为

**现象**：
```bash
publish-toolkit publish --no-git-check
# options.gitCheck 为 false（预期）
publish-toolkit publish
# options.gitCheck 为 undefined（非 false）
```

**原因**：
Commander 的 `--no-*` 标志在未提供时返回 `undefined`，而非 `false`。

**解决方案**：
```typescript
skipGitCheck: options.gitCheck === false,
```
显式比较 `=== false`，而非依赖真值判断。

### 坑 3：临时 .npmrc 未清理

**现象**：
脚本异常退出后，`.npmrc` 残留，包含敏感 token。

**原因**：
未注册进程退出清理钩子。

**解决方案**：
```typescript
process.on("exit", cleanupNpmrc);
process.on("SIGINT", () => { cleanupNpmrc(); process.exit(130); });
process.on("SIGTERM", () => { cleanupNpmrc(); process.exit(143); });
```

### 坑 4：Monorepo 中 bin 链接问题

**现象**：
```bash
npx publish-toolkit
# 报错：command not found
```

**原因**：
pnpm workspace 包默认不将 bin 链接到根目录的 `node_modules/.bin`。

**解决方案**：
- 在根 package.json 的 devDependencies 中添加 `"@usethink/publish-toolkit": "workspace:*"`
- 重新执行 `pnpm install`
- 或使用 `pnpm --filter @usethink/publish-toolkit exec publish-toolkit`

### 坑 5：Git 状态检查包含无关文件

**现象**：
```bash
publish-toolkit publish
# 报错：工作区有未提交的变更
# 实际是 .qwen/ 或 docs/ 的修改
```

**原因**：
`git status --porcelain` 检查整个仓库，包含非项目文件。

**解决方案**：
- 提供 `--no-git-check` 参数跳过检查
- 或在发布前清理工作区
- 后续可优化为仅检查 package.json 所在目录

### 坑 6：TypeScript 编译输出路径

**现象**：
```bash
node dist/cli.js
# 报错：Cannot find module './publish/publisher.js'
```

**原因**：
TypeScript `tsconfig.json` 中 `outDir` 配置为 `dist`，但 import 语句使用相对路径 `.js` 扩展名（ESM 模块解析需要）。

**解决方案**：
保持 `moduleResolution: "bundler"`，import 时保留 `.js` 扩展名，确保编译后路径正确。

## 8. 注意事项

### 8.1 环境变量

| 变量 | 必需 | 说明 |
|------|------|------|
| `NPM_TOKEN` | publish 命令必需 | npm 自动化 access token |
| `NODE_AUTH_TOKEN` | CI 可选 | npm publish 自动读取，与 NPM_TOKEN 二选一 |

**安全提示**：
- 永远不要将 `NPM_TOKEN` 提交到 git
- 使用 npm 的 "Automation" 类型 token（无需 2FA）
- CI 中通过 GitHub Secrets / GitLab CI Variables 注入

### 8.2 Git 检查

- 默认执行 `git status --porcelain` 检查工作区
- 有未提交变更时拒绝发布（防止发布未完成代码）
- 可通过 `--no-git-check` 跳过（不推荐）

### 8.3 版本检查

- 默认检查 npm registry 上版本是否已存在
- 防止重复发布相同版本
- 可通过 `--no-version-check` 跳过

### 8.4 Registry 配置

- 默认 registry：`https://registry.npmjs.org/`
- 支持自定义 registry（如 GitHub Packages）
- 临时写入 `.npmrc`，发布完成后自动清理

### 8.5 混淆引擎

- 当前为占位实现，**不实际执行混淆**
- 接入真实混淆器（如 `javascript-obfuscator`）时：
  - 替换 `src/obfuscate/obfuscator.ts` 中的 `obfuscate` 方法
  - 添加 `javascript-obfuscator` 到 dependencies
  - 保持输入输出接口不变

### 8.6 发布权限

- Scoped 包（`@usethink/*`）需要 `--access public`
- 私有包使用 `--access restricted`
- 403 错误通常表示 token 权限不足或 scope 不匹配

### 8.7 pnpm vs npm

- 本地开发优先使用 `pnpm exec`
- CI 中可自由选择 `pnpm` 或 `npm`
- 工具会自动检测可用的包管理器

## 11. 发布到 npm 完整指南

### 11.1 前置条件

在发布到 npm 之前，需要完成以下准备：

1. **npm 账号**：确保拥有 `@usethink` scope 的发布权限
2. **GitHub 仓库**：已创建 `qwdingyu/publish-toolkit` 公开仓库
3. **npm Token**：创建 Automation 类型的 access token（无需 2FA）

### 11.2 获取 npm Token

```bash
# 1. 登录 npm（交互式，需要输入用户名和密码）
npm login

# 2. 创建 Automation token
# 访问 https://www.npmjs.com/settings/tokens
# 点击 "Generate New Token" → "Classic Token" → "Automation"
# 复制生成的 token（格式：npm_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx）
```

### 11.3 配置 GitHub Secret

```bash
# 在 publish-toolkit 仓库中设置 NPM_TOKEN secret
cd tools/publish-toolkit
gh secret set NPM_TOKEN --repo qwdingyu/publish-toolkit

# 或使用环境变量方式
NPM_TOKEN="npm_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx" \
  gh secret set NPM_TOKEN --repo qwdingyu/publish-toolkit
```

### 11.4 本地首次发布（推荐）

在本地完成首次发布，验证包结构：

```bash
cd tools/publish-toolkit

# 1. 登录 npm
npm login

# 2. 干跑测试（不实际发布）
npm publish --dry-run

# 3. 正式发布
npm publish --access public
```

发布成功后，包将可在 npmjs.com 上查看：
https://www.npmjs.com/package/@usethink/publish-toolkit

### 11.5 后续版本发布流程

#### 方式一：Git Tag 触发（推荐）

```bash
# 1. 修改版本号
# 编辑 tools/publish-toolkit/package.json 中的 version 字段
# 例如：0.1.0 → 0.1.1

# 2. 提交变更
git add -A
git commit -m "chore: bump version to 0.1.1"

# 3. 打 tag
git tag v0.1.1

# 4. 推送代码和 tag
git push origin main
git push origin v0.1.1
```

### 11.7 版本管理规范

遵循语义化版本控制（Semantic Versioning）：

- **MAJOR**：不兼容的 API 修改
- **MINOR**：向下兼容的功能新增
- **PATCH**：向下兼容的问题修复

版本号示例：
- `0.1.0` → `0.1.1`：bug 修复
- `0.1.0` → `0.2.0`：新功能
- `0.1.0` → `1.0.0`：重大变更

### 11.8 常见问题

#### Q: 发布时提示 403 Forbidden

**原因**：npm token 权限不足或 scope 不匹配

**解决**：
1. 确认 token 有 `@usethink` scope 的发布权限
2. 或使用 `--access public` 参数（scoped 包必需）

#### Q: 发布时提示 404 Not Found

**原因**：包名已存在或 registry 配置错误

**解决**：
1. 检查包名是否已存在于 npm
2. 确认 registry 为 `https://registry.npmjs.org/`

#### Q: GitHub Actions 无法访问 npm

**原因**：NPM_TOKEN secret 未配置

**解决**：
```bash
gh secret set NPM_TOKEN --repo qwdingyu/publish-toolkit
```

#### Q: 如何在本地开发时测试未发布版本

