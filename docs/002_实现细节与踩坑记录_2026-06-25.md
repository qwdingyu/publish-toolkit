## 2. 发布引擎实现细节

### 2.1 核心流程（实际代码）

```
PublishToolkit.publish(npmToken)
  │
  ├─ Step 1: 验证参数
  │   ├─ dry-run 模式允许 NPM_TOKEN 为空
  │   ├─ 检查 package.json 存在且非 private
  │   └─ 读取包名和版本
  │
  ├─ Step 2: git 检查（可跳过）
  │   ├─ git rev-parse --show-toplevel
  │   ├─ git status --porcelain
  │   └─ 有未提交变更则报错退出
  │
  ├─ Step 3: 版本检查（可跳过）
  │   ├─ npm view <pkg> version --registry=<registry>
  │   └─ 与本地 package.json 比较
  │
  ├─ Step 4: 构建
  │   ├─ 检测 prepack 脚本 → 跳过（npm publish 自动执行）
  │   ├─ 检测 build 脚本 → 执行构建
  │   │   ├─ 优先检测 pnpm（which pnpm）
  │   │   ├─ pnpm 失败则回退 npm
  │   │   └─ dry-run 模式仅打印不执行
  │   └─ 无脚本 → 跳过
  │
  ├─ Step 5: 配置认证
  │   ├─ dry-run 模式：仅打印将要写入的内容，不写文件
  │   └─ 正式模式：
  │       ├─ 写入临时 .npmrc（registry + _authToken）
  │       ├─ 备份原有 .npmrc
  │       └─ 注册进程退出清理钩子（exit/SIGINT/SIGTERM）
  │
  └─ Step 6: 发布
      ├─ 检测 pnpm/npm
      ├─ pnpm 添加 --no-git-checks 参数
      ├─ dry-run 添加 --dry-run 参数
      ├─ 执行 npm publish
      └─ 错误解析（404/403/401/EOTP）
```

### 2.2 关键实现细节

#### 2.2.1 动态版本读取

```typescript
// src/cli.ts
const pkgVersion = JSON.parse(
  readFileSync(new URL("../package.json", import.meta.url), "utf-8")
).version;
```

**设计意图**：避免 CLI 版本与 package.json 不一致。之前硬编码版本导致发布了错误的版本号。

#### 2.2.2 pnpm 检测与回退

```typescript
// src/publish/publisher.ts
const npmBin = tryRun("which pnpm 2>/dev/null", pkgDir) ? "pnpm" : "npm";

// 构建时 pnpm 失败回退 npm
if (npmBin === "pnpm") {
  warn("  pnpm 构建失败，尝试 npm...");
  run("npm run build", pkgDir, { verbose: opts.verbose });
}

// 发布时 pnpm 添加 --no-git-checks
if (npmBin === "pnpm") pubArgs.push("--no-git-checks");
```

**设计意图**：兼容不同项目使用的包管理器，确保在任何环境下都能成功构建和发布。

#### 2.2.3 跨平台 pnpm 检测

```typescript
// src/publish/publisher.ts
function detectPnpm(cwd: string): boolean {
  const isWin = process.platform === "win32";
  const cmd = isWin ? "where pnpm" : "command -v pnpm";
  const result = tryRun(cmd, cwd);
  return result.success && result.stdout.trim().length > 0;
}
```

**踩坑原因**：原实现使用 `which pnpm`，在 Windows 上不存在 `which` 命令，导致检测失败。

**解决方案**：Unix 使用 `command -v`，Windows 使用 `where`，通过 `tryRun` 统一处理跨平台差异。

#### 2.2.4 命令执行健壮性

```typescript
interface CommandResult {
  success: boolean;
  stdout: string;
  stderr: string;
  code: number | null;
}

function run(cmd: string, cwd: string, opts?: RunOptions): string {
  const options = { cwd, encoding: "utf-8" as const, timeout: 30_000, ...opts };
  if (opts?.verbose) console.log(`  $ ${cmd}`);
  try {
    const stdout = execSync(cmd, { ...options, stdio: "pipe" });
    const out = (stdout || "").toString().trim();
    if (opts?.verbose && out) console.log(`  => ${out}`);
    return out;
  } catch (err) {
    const execErr = err as Error & { status?: number; stderr?: Buffer; stdout?: Buffer };
    const stderr = (execErr.stderr || execErr.stdout || Buffer.from("")).toString("utf-8").trim();
    throw new Error(stderr || `Command failed: ${cmd} (exit ${execErr.status ?? 1})`);
  }
}

function tryRun(cmd: string, cwd: string, opts?: RunOptions): CommandResult {
  try {
    const stdout = run(cmd, cwd, opts);
    return { success: true, stdout, stderr: "", code: 0 };
  } catch (err) {
    const execErr = err as Error & { status?: number; stderr?: Buffer; stdout?: Buffer };
    const stderr = (execErr.stderr || execErr.stdout || Buffer.from("")).toString("utf-8").trim();
    return { success: false, stdout: "", stderr, code: execErr.status ?? 1 };
  }
}
```

**踩坑原因**：原 `tryRun` 直接返回字符串，无法区分"命令成功但无输出"和"命令失败"。

**解决方案**：返回结构化结果 `CommandResult`，调用方可根据 `success` 字段判断执行状态，保留 `stderr` 和 `code` 用于诊断。

#### 2.2.5 npm view 超时优化

```typescript
const publishedResult = tryRun(
  `npm view ${pkgName} version --registry=${opts.registry} 2>/dev/null || true`,
  pkgDir,
  { timeout: 60_000 }
);
```

**踩坑原因**：CI 环境网络抖动时，30s 超时可能导致误判为"版本首次发布"。

**解决方案**：将 `npm view` 超时提升到 60s，并增加 verbose 日志输出失败原因，便于排查网络问题。

#### 2.2.6 package.json 解析安全

```typescript
let pkg: {
  name: string;
  version: string;
  private?: boolean;
  scripts?: Record<string, string>;
};
try {
  pkg = JSON.parse(readFileSync(pkgJsonPath, "utf-8"));
} catch (err) {
  error(`package.json 解析失败: ${(err as Error).message}`);
  return this.fail("", "", "package.json 格式错误");
}
```

**踩坑原因**：裸 `JSON.parse` 在 package.json 格式错误时会抛出未捕获异常，导致进程崩溃。

**解决方案**：增加 try-catch，返回友好的错误信息，避免进程崩溃。

#### 2.2.7 构建失败错误详情保留

```typescript
// pnpm 构建失败时保留原始错误信息
if (npmBin === "pnpm") {
  const pnpmErr = (err as Error).message;
  warn(`  pnpm 构建失败: ${pnpmErr}，尝试 npm...`);
  try {
    run("npm run build", pkgDir, { verbose: opts.verbose });
    startStep("构建").done();
  } catch (err2) {
    const npmErr = (err2 as Error).message;
    error(`构建失败: ${npmErr}`);
    return this.fail(pkgName, pkgVersion, `构建失败: ${npmErr}`);
  }
}
```

**踩坑原因**：原实现构建失败时返回通用错误信息，丢失了原始错误上下文。

**解决方案**：保留 pnpm/npm 的原始错误消息，便于快速定位构建问题。

#### 2.2.3 dry-run 完整跳过

```typescript
// Step 4: 构建
if (opts.dryRun) {
  log(`  [dry-run] 将执行构建: ${npmBin} run build`);
  // 不执行实际构建
}

// Step 5: .npmrc
if (!opts.dryRun) {
  // 写入 .npmrc
} else {
  log(`  [dry-run] 将写入 .npmrc 到: ${pkgDir}`);
  // 不写文件
}

// Step 6: 发布
if (opts.dryRun) {
  log(`  [dry-run] 将执行: ${npmBin} ${pubArgs.join(" ")}`);
  // 不执行 publish
}
```

**设计意图**：dry-run 应该是真正的"预览"，不产生任何副作用。

#### 2.2.4 .npmrc 安全管理

```typescript
function registryHost(registry: string): string {
  const u = new URL(registry);
  return `//${u.host}${u.pathname.replace(/\/+$/, "")}`;
}

let _rcPath: string | null = null;
let _rcBackup: string | null = null;

function cleanupNpmrc() {
  if (!_rcPath) return;
  try { unlinkSync(_rcPath); } catch {}
  if (_rcBackup !== null) writeFileSync(_rcPath, _rcBackup, "utf-8");
}

// 注册清理钩子
process.on("exit", cleanupNpmrc);
process.on("SIGINT", () => { cleanupNpmrc(); process.exit(130); });
process.on("SIGTERM", () => { cleanupNpmrc(); process.exit(143); });
```

**设计意图**：
- 将 registry 转为 npm 的 `//host/path` 形式写入 `_authToken`，兼容私有 registry。
- dry-run 模式完全跳过 `.npmrc` 写入，不会产生残留。
- 备份并恢复原有 `.npmrc`，避免污染开发环境。

### 2.3 混淆引擎实现细节

#### 2.3.1 已实现功能

混淆引擎**不是占位实现**，而是完整实现了基于 `javascript-obfuscator` 的混淆功能：

- **四级混淆策略**：none / light / medium / aggressive
- **文件遍历**：递归扫描 .js/.mjs/.ts/.cjs/.d.ts/.jsx/.tsx
- **排除规则**：支持 glob 模式（`*.ext`、`**/*.ext`、`dir/**`）
- **Source Map**：可选生成 separate source map
- **混淆报告**：可选生成 JSON 格式报告（obfuscate-report.json）
- **零侵入**：输入输出均为文件路径，不修改项目源码

#### 2.3.2 ESM/CJS 兼容处理

```typescript
// javascript-obfuscator 的 ESM/CJS 类型声明不一致：
// 运行时 default export 是函数，但类型文件把 obfuscate 声明为 named export
const obfuscate = obfuscateModule as unknown as {
  obfuscate(sourceCode: string, inputOptions?: ObfuscatorOptions): ObfuscationResult;
};
```

**踩坑原因**：`javascript-obfuscator` 的 type 声明使用 named export，但实际运行时是 default export。

**解决方案**：运行时类型断言，不改变调用方式。

#### 2.3.3 级别预设配置

| 级别 | 核心配置 | 适用场景 |
|------|----------|----------|
| none | compact: false, 无其他混淆 | 调试或不需要保护 |
| light | compact: true, stringArray: RC4 | 基础保护，体积增加小 |
| medium | + controlFlowFlattening, deadCodeInjection, transformObjectKeys | 大多数发布场景 |
| aggressive | + debugProtection, disableConsoleOutput, unicodeEscapeSequence | 极致保护，体积增加大 |

## 4. 踩坑记录与解决方案

### 坑 1：npx 在某些环境被解析为 npm run

**现象**：
```bash
npx publish-toolkit publish
# 报错：Missing script: "publish-toolkit"
```

**原因**：
在某些 shell/目录配置下，`npx` 被解析为 `npm run`，而不是执行 `node_modules/.bin` 下的二进制。

**解决方案**：
- 优先使用 `pnpm exec publish-toolkit publish`（项目已安装为 devDependency 时）
- 退回到 `npx` 时确保工具未安装在本地

**验证**：
```bash
pnpm exec publish-toolkit --version  # 正常
npx publish-toolkit --version        # 某些环境可能失败
```

### 坑 2：Commander 的 --no-* 标志行为

**现象**：
```bash
publish-toolkit publish --no-git-check
# options.gitCheck 为 false（预期）

publish-toolkit publish
# options.gitCheck 为 undefined（非 false）
```

**原因**：
Commander 的 `--no-*` 标志在未提供时返回 `undefined`，而非 `false`。

**解决方案**：
```typescript
skipGitCheck: options.gitCheck === false,
```
显式比较 `=== false`，而非依赖真值判断。

### 坑 3：临时 .npmrc 未清理

**现象**：
脚本异常退出后，`.npmrc` 残留，包含敏感 token。

**原因**：
未注册进程退出清理钩子。

**解决方案**：
```typescript
process.on("exit", cleanupNpmrc);
process.on("SIGINT", () => { cleanupNpmrc(); process.exit(130); });
process.on("SIGTERM", () => { cleanupNpmrc(); process.exit(143); });
```

**注意**：dry-run 模式现在完全跳过 .npmrc 写入，不会产生残留。

### 坑 4：javascript-obfuscator ESM/CJS 类型不匹配

**现象**：
```typescript
import obfuscateModule from "javascript-obfuscator";
// TypeScript 报错：类型不匹配
```

**原因**：
`javascript-obfuscator` 的 type 声明使用 named export，但实际运行时是 default export。

**解决方案**：
```typescript
const obfuscate = obfuscateModule as unknown as {
  obfuscate(sourceCode: string, inputOptions?: ObfuscatorOptions): ObfuscationResult;
};
```

运行时类型断言，不改变调用方式。

### 坑 5：Git 状态检查包含无关文件

**现象**：
```bash
publish-toolkit publish
# 报错：工作区有未提交的变更
# 实际是 .qwen/ 或 docs/ 的修改
```

**原因**：
`git status --porcelain` 检查整个仓库，包含非项目文件。

**解决方案**：
- 提供 `--no-git-check` 参数跳过检查
- 或在发布前清理工作区

**后续优化方向**：仅检查 package.json 所在目录，而非整个仓库。

### 坑 6：dry-run 模式仍有副作用

**现象**：
```bash
publish-toolkit publish --dry-run
# 1. 实际执行了构建命令
# 2. 写入了 .npmrc 文件
```

**原因**：
dry-run 标志仅在最后一步发布时检查，前面的构建和 .npmrc 步骤未区分。

**解决方案（v0.1.8）**：
```typescript
// Step 4: 构建
if (opts.dryRun) {
  log(`  [dry-run] 将执行构建: ${npmBin} run build`);
  // 不执行实际构建
}

// Step 5: .npmrc
if (!opts.dryRun) {
  // 写入 .npmrc
} else {
  log(`  [dry-run] 将写入 .npmrc 到: ${pkgDir}`);
  // 不写文件
}
```

### 坑 7：版本号硬编码导致发布错误版本

**现象**：
CLI 中硬编码版本为 `0.1.4`，但实际发布时 package.json 已经是 `0.1.6`。

**原因**：
`cli.ts` 中手动维护版本号，与 package.json 不同步。

**解决方案（v0.1.7）**：
```typescript
const pkgVersion = JSON.parse(
  readFileSync(new URL("../package.json", import.meta.url), "utf-8")
).version;
```

动态读取 package.json，确保版本一致。

### 坑 8：CI 中 npm install 修改 lockfile

**现象**：
GitHub Actions 中 `npm install` 会修改 package-lock.json，导致 git check 失败。

**原因**：
`npm install` 在某些情况下会更新 lockfile，即使没有依赖变化。

**解决方案**：
在 CI 中使用 `npm ci` 代替 `npm install`：
```yaml
- name: Install dependencies
  run: npm ci
```

`npm ci` 严格按 lockfile 安装，不会修改 lockfile。

### 坑 9：pnpm 检测使用 which 命令

**现象**：
在 Windows 或某些精简容器中 `which` 命令不存在。

**原因**：
`which pnpm 2>/dev/null` 在非 Unix 系统上可能失败。

**现状**：
当前使用 `tryRun` 捕获错误，失败时回退到 npm。在 Windows 上可能需要额外处理。

**迁移步骤**：
1. 将 publish-toolkit 从 cf-core/tools/ 移动到独立仓库 `/Users/dingyuwang/0-X/node/publish-toolkit/`
2. 发布到 npmjs（@usethink/publish-toolkit）
3. 更新 cf-core package.json，将 workspace 依赖改为 npm 依赖
4. 更新 GitHub Actions workflow，使用 `npx publish-toolkit publish`
5. 清理 cf-core 中的旧 scripts/ 和 tools/ 目录

### 6.2 cf-core 的 package.json 配置

```json
{
  "devDependencies": {
    "@usethink/publish-toolkit": "^0.1.8"
  },
  "scripts": {
    "prepublishOnly": "npm run type-check && npm test",
    "prepack": "npm run build"
  }
}
```

**注意**：
- `prepublishOnly` 在 npm publish 前自动执行
- `prepack` 在 npm pack 前自动执行构建
- publish-toolkit 会检测 prepack 脚本并跳过手动构建

### 6.3 cf-core 的 GitHub Actions 配置

```yaml
name: Publish

on:
  push:
    tags:
      - 'v*'
  workflow_dispatch:
    inputs:
      dry_run:
        description: 'Dry-run 模式（只预览，不实际发布）'
        type: boolean
        default: false
      tag:
        description: 'dist-tag（正式版用 latest，beta 用 beta）'
        type: string
        default: latest

jobs:
  publish:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 22
          registry-url: 'https://registry.npmjs.org'

      - run: npm ci          # 使用 npm ci 避免 lockfile 被修改
      - run: npm run type-check
      - run: npm test
      - run: npm run build

      - run: pnpm exec publish-toolkit publish --registry=https://registry.npmjs.org/ --verbose
        env:
          NPM_TOKEN: ${{ secrets.NPM_TOKEN }}
```

**关键点**：
- 使用 `npm ci` 确保 lockfile 不被修改
- 使用 `pnpm exec publish-toolkit` 避免 `npx` 被解析为 `npm run` 的风险
- tag 触发自动发布，workflow_dispatch 支持手动触发和 dry-run
- NPM_TOKEN 通过 GitHub Secrets 注入

1. **本仓库自发布未使用 toolkit CLI**：`.github/workflows/publish.yml` 当前直接 `npm publish`，不走 publish 引擎
2. **混淆引擎未集成到发布流程**：obfuscate 命令独立，发布流程中未自动调用
3. **Git 检查范围过大**：检查整个仓库而非 package.json 所在目录
4. **pnpm 检测使用 which**：Windows 兼容性未验证
5. **无 changelog 自动生成**：需要手动维护版本说明
