### 1.2 发布引擎关键类型

```ts
export type PackageManager = "npm" | "pnpm" | "auto";

export interface PublishOptions {
  packageDir?: string;
  registry?: string;
  dryRun?: boolean;
  tag?: string;
  otp?: string;
  access?: string;
  skipGitCheck?: boolean;
  skipVersionCheck?: boolean;
  verbose?: boolean;
  packageManager?: PackageManager;
}
```

注释意图：

- `packageManager` 是发布命令选择，不是 CLI 启动方式。
- 默认值是 `npm`。
- `auto` 只根据项目文件判断，不根据全局命令判断。

### 1.3 包管理器解析实现

当前逻辑：

```ts
function resolvePackageManager(
  pkgDir: string,
  pkg: { packageManager?: string },
  configured: PackageManager
): { bin: "npm" | "pnpm"; reason: string } {
  if (configured === "npm" || configured === "pnpm") {
    return { bin: configured, reason: "显式配置" };
  }

  if (pkg.packageManager?.startsWith("pnpm@")) {
    return { bin: "pnpm", reason: "package.json packageManager" };
  }

  if (existsSync(resolve(pkgDir, "pnpm-lock.yaml")) && !existsSync(resolve(pkgDir, "package-lock.json"))) {
    return { bin: "pnpm", reason: "pnpm-lock.yaml" };
  }

  return { bin: "npm", reason: "auto 默认" };
}
```

实现注释：

- 显式参数优先级最高。
- `packageManager: "pnpm@..."` 是项目维护者的明确声明。
- 仅存在 `pnpm-lock.yaml` 且不存在 `package-lock.json` 时，才在 auto 模式选择 pnpm。
- 同时存在 `package-lock.json` 时，auto 不猜测 pnpm，回到 npm。
- 不再执行 `which pnpm` / `command -v pnpm` / `where pnpm`。

### 1.4 CLI 参数传递

`src/cli.ts` 中 publish 命令暴露：

```ts
.option("--package-manager <name>", "发布命令使用的包管理器: npm | pnpm | auto", "npm")
```

并在 CLI 层校验枚举值：

```ts
if (!isPackageManager(options.packageManager)) {
  console.error(`错误: --package-manager 只能是 npm、pnpm 或 auto，当前值: ${options.packageManager}`);
  process.exit(1);
}
```

注意：

- commander 不会自动校验枚举值。
- CLI 和 `PublishToolkit` 构造函数都做运行时校验，非法值直接失败。

### 1.5 dry-run 语义

dry-run 会：

- 允许缺少 `NPM_TOKEN`。
- 跳过真实构建命令。
- 跳过真实 npm userconfig 写入。
- 跳过真实发布命令。
- 打印将执行的构建与发布命令。

dry-run 不会：

- 修改 package。
- 写 token。
- 发布 npm 包。

### 1.6 认证与临时 npm userconfig 清理

正式发布时会写入系统临时目录中的 npm userconfig：

```text
registry=https://registry.npmjs.org/
//registry.npmjs.org/:_authToken=<token>
```

然后通过环境变量传给发布命令：

```text
NPM_CONFIG_USERCONFIG=/tmp/publish-toolkit-xxxx/.npmrc
```

清理钩子：

```ts
process.on("exit", cleanupNpmrc);
process.on("SIGINT", () => { cleanupNpmrc(); process.exit(130); });
process.on("SIGTERM", () => { cleanupNpmrc(); process.exit(143); });
```

注意：

- dry-run 不写 userconfig。
- 不再写项目目录 `.npmrc`，避免污染仓库或误提交 token。
- 退出时删除整个临时目录。
- 日志中只显示 token 是否存在，不输出 token 内容。

## 2. 本次事故复盘：mach-calculators 触发 pnpm ignored builds

### 2.1 背景

`@usethink/mach-calculators` 是从 monorepo 中迁出的独立 npm 包，用户明确要求：

- 包名：`@usethink/mach-calculators`
- 位置：`/Users/dingyuwang/0-X/2-node/mach-calculators`
- 发布方式：使用 npm，不要使用 pnpm。

迁移初期尝试沿用 `@usethink/cf-core` 的发布方式，即使用 `@usethink/publish-toolkit`。

### 2.2 现象

GitHub Actions 失败：

```text
ERR_PNPM_IGNORED_BUILDS Ignored build scripts: esbuild@0.28.1
Run "pnpm approve-builds" to pick which dependencies should be allowed to run scripts.
```

当时 Actions 中可见：

```text
[publish] 执行: pnpm publish --registry=https://registry.npmjs.org/ --access=public --tag=latest --no-git-checks
```

这和用户要求的 npm 发布矛盾。

### 2.3 为什么 cf-core 正常

`@usethink/cf-core` 也使用 publish-toolkit，并且当时同样执行 `pnpm publish`。

它能成功，是因为 cf-core 已经配置：

```text
onlyBuiltDependencies[]=esbuild
```

或在 pnpm workspace 配置中允许 `esbuild` postinstall。

因此 cf-core 成功不是因为 publish-toolkit 没用 pnpm，而是 cf-core 对 pnpm v11 的 build script 策略做了额外适配。

### 2.4 根因

旧版 publish-toolkit 使用全局命令探测决定包管理器：

```ts
detectPnpm(pkgDir) ? "pnpm" : "npm"
```

这有两个错误假设：

1. runner 上安装了 pnpm，就代表项目应该用 pnpm 发布。
2. pnpm 可执行，就代表项目已经配置好 pnpm v11 的 build scripts 白名单。

这两个假设都不成立。

### 2.5 修复方案

修复为：

- 默认 `--package-manager npm`。
- 显式 `--package-manager pnpm` 时才使用 pnpm。
- `--package-manager auto` 只看项目元数据，不看全局环境。
- dry-run 日志明确打印：

```text
包管理器: npm（显式配置）
[dry-run] 将执行: npm publish ...
```

### 2.6 回归测试

新增测试覆盖：

1. 默认使用 npm，不能因为环境中存在 pnpm 就切换发布命令。
2. 显式指定 pnpm 时才使用 pnpm publish。
3. auto 模式只根据项目元数据选择 pnpm。

测试文件：`tests/publisher.test.ts`

## 3. 其他踩坑记录

### 坑 1：npm scoped 包发布返回 E404，不一定是包不存在

现象：

```text
npm error 404 Not Found - PUT https://registry.npmjs.org/@usethink%2fmach-calculators
```

排查：

- GitHub Actions 中 `NPM_TOKEN` 已设置，publish-toolkit Step 1 显示 token 存在。
- publish-toolkit Step 3 显示 `@usethink/mach-calculators@0.1.3` 尚未发布。
- publish-toolkit Step 6 执行的是 `npm publish --registry=https://registry.npmjs.org/ --access=public --tag=latest`。
- npm 仍返回 `404 Not Found - PUT https://registry.npmjs.org/@usethink%2fmach-calculators`。

结论：

- 如果已经有 `--access public`，这个 E404 通常不是包内容问题，而是 npm 对无权限 scoped package 的隐藏式错误。
- 不能只看 token 字符串形态判断有效性，必须用 `npm whoami` 验证。

完整排查命令：

```bash
tmpdir=$(mktemp -d)
trap 'rm -rf "$tmpdir"' EXIT
umask 077
printf '%s\n' '//registry.npmjs.org/:_authToken=<NPM_TOKEN>' > "$tmpdir/.npmrc"
npm whoami --registry=https://registry.npmjs.org/ --userconfig="$tmpdir/.npmrc"
```

判断：

- 返回用户名：token 至少能被 npm registry 识别。
- 返回 `E401 Unauthorized`：token 无效、已撤销、复制不完整，或不是 npm registry token。

继续检查权限：

```bash
npm access list packages @usethink --registry=https://registry.npmjs.org/ --json
```

目标包应显示：

```json
{
  "@usethink/mach-calculators": "read-write"
}
```

本次事故中的实际结论：

- 用户提供的 token 写入 GitHub Secret 后仍失败。
- 使用同一个 token 在本机临时 userconfig 下执行 `npm whoami` 返回 `E401 Unauthorized`。
- 本机当前 npm 登录态 `npm whoami` 返回 `iusethink`。
- `npm access list packages @usethink --json` 显示 `@usethink/mach-calculators` 为 `read-write`。
- 因此问题不是 GitHub Secret 写入失败，而是用户给出的 token 无效。

最终解决：

```bash
awk -F'_authToken=' '/registry\.npmjs\.org.*_authToken=/{print $2; exit}' ~/.npmrc \
  | gh secret set NPM_TOKEN --repo qwdingyu/mach-calculators
```

然后 rerun GitHub Actions：

```bash
gh run rerun <run-id>
gh run view <run-id> --json status,conclusion,url
```

验证结果：

- Actions 日志显示 `NPM_TOKEN 已设置`。
- `@usethink/mach-calculators@0.1.3` 被判断为尚未发布。
- `publish-toolkit publish --package-manager npm` 发布成功。
- `npm view @usethink/mach-calculators version dist-tags --json` 显示 `latest: 0.1.3`。

注意事项：

- 不要把 token 写进仓库文件。
- 不要把 token 放在命令行参数里提交到 issue/文档/日志。
- `printf '%s' '<token>' | gh secret set ...` 比把 token 放到命令参数里更不容易泄漏到 shell history。
- 若从 `~/.npmrc` 同步 token，必须先确认本机登录账号和 package access 是正确的。

### 坑 2：手工发布后再推 tag 会触发重复发布

现象：

手工发布 `@usethink/publish-toolkit@0.1.15` 后，再推送 `v0.1.15` tag，GitHub Actions 自动执行 `npm publish`，失败：

```text
403 Forbidden - You cannot publish over the previously published versions: 0.1.15.
```

原因：

- npm 不允许覆盖同名同版本。
- tag workflow 没有先检查版本是否已存在。

解决方案：

workflow 增加 `Check published version` 步骤：

```bash
if npm view "${PACKAGE_NAME}@${PACKAGE_VERSION}" version --registry=https://registry.npmjs.org/ >/dev/null 2>&1; then
  echo "exists=true" >> $GITHUB_OUTPUT
  echo "${PACKAGE_NAME}@${PACKAGE_VERSION} 已存在，跳过发布。"
else
  echo "exists=false" >> $GITHUB_OUTPUT
fi
```

`Publish to npm` step 加条件：

```yaml
if: steps.published.outputs.exists != 'true'
```

结果：

- 版本已存在时 workflow 成功退出。
- 手工发布 + tag 追溯可以共存。

### 坑 3：package.json 与 package-lock.json 版本不一致

现象：

- `package.json` 已经是 `0.1.13`。
- `package-lock.json` 仍记录 `0.1.5`。

风险：

- CI/npm 元数据不一致。
- 审计和安装过程容易产生误判。

解决方案：

```bash
npm install --package-lock-only
```

当前已同步到 `0.1.15`。

### 坑 4：dist 被 .gitignore 忽略但仍需同步已跟踪文件

现象：

构建后执行：

```bash
git add dist/...
```

报错：

```text
The following paths are ignored by one of your .gitignore files: dist
```

原因：

- `dist/` 被 `.gitignore` 忽略。
- 但仓库中已有部分 `dist` 文件是已跟踪文件，修改后仍需要提交。

解决方案：

```bash
git add -u
```

对已跟踪的 `dist` 文件收录修改即可。不要强行添加新的未跟踪 dist 文件，除非确认需要发布。

### 坑 5：npx 与 pnpm exec 只是 CLI 启动方式

现象：

文档中曾经把 `pnpm exec publish-toolkit` 写成推荐发布方式，容易让人误解为内部发布命令也应使用 pnpm。

结论：

- `npx --yes -p @usethink/publish-toolkit publish-toolkit publish`：不安装到项目，临时下载 scoped 包并启动 CLI。
- `npx publish-toolkit publish`：项目已安装 `@usethink/publish-toolkit` 为 devDependency 时，调用本地 bin。
- `pnpm exec publish-toolkit publish`：用 pnpm 启动 CLI。
- 二者默认都会在 CLI 内部执行 `npm publish`。
- 内部发布命令由 `--package-manager` 决定。

### 坑 6：prepack 与 build 的职责边界

现象：

一些项目已有：

```json
{
  "scripts": {
    "prepack": "npm run build"
  }
}
```

如果 publish-toolkit 在 Step 4 也执行 build，会造成重复构建。

当前策略：

- 有 `prepack` 时，发布引擎跳过手动构建。
- 让 `npm publish` 或 `pnpm publish` 自己触发生命周期脚本。

注意：

- dry-run 中仍会打印 prepack 已定义。
- 如果没有 prepack 但有 build，才由 publish-toolkit 执行 build。

### 坑 7：临时 npm userconfig 不能残留 token

风险：

- 发布过程需要写 registry token。
- 异常退出可能留下 token 文件。

当前策略：

- 使用 `mkdtempSync(join(tmpdir(), "publish-toolkit-"))` 创建临时目录。
- 写入临时 `.npmrc`。
- 通过 `NPM_CONFIG_USERCONFIG` 传给 npm。
- 退出时删除整个临时目录。
- SIGINT/SIGTERM 均清理。

### 坑 8：npm audit fix --force 不应自动执行

现象：

执行 lockfile 同步后 npm 提示存在 audit 漏洞。

决策：

- 不自动执行 `npm audit fix --force`。
- 原因是 `--force` 可能引入破坏性依赖升级。
- 漏洞处理应独立评估，尤其是 `javascript-obfuscator` 依赖链。

## 6. 当前公开契约

对消费项目承诺：

- 默认发布命令是 npm。
- pnpm 发布必须显式选择或通过项目元数据选择。
- dry-run 不写 token、不发布包。
- 正式发布会写临时 npm userconfig 并清理。
- 版本已存在时 publish 引擎默认失败，防止覆盖。

对 publish-toolkit 自身维护承诺：

- 自发布 workflow 直接使用 npm。
- workflow 会跳过已存在版本，避免重复发布失败。
- tag 用于源码追溯，不应强制覆盖 npm 已有版本。
